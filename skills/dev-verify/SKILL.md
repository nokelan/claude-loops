# /dev-verify — 최종 검증 & 품질 심사

SPEC.md의 모든 완료 기준을 최종 점검하고, 코드 품질과 목표 정합성을 심사한다.

## 핵심 원칙

- /dev-loop가 끝난 후 독립적 관점에서 재검증하는 단계
- 구현자와 검증자를 분리 — 이 스킬은 구현하지 않고 오직 검증만 한다
- 결과는 PASS / CONDITIONAL PASS / FAIL 세 가지로만 판정

## 실행 흐름

### Phase 0: 사전 조건 확인

1. SPEC.md 존재 여부 확인
2. 코드/결과물 존재 여부 확인
3. 둘 다 없으면 중단

### Phase 1: AC 체크리스트 검증

SPEC.md의 완료 기준(AC)을 하나씩 독립적으로 검증한다.

각 AC에 대해:
- 코드/결과물을 직접 확인하여 기준 충족 여부 판단
- 가능하면 실행하여 눈으로 확인
- 판정: `[PASS]` / `[FAIL]` / `[PARTIAL]`

### Phase 2: 코드 품질 심사 (5점 척도)

| 항목 | 체크 |
|------|------|
| 빌드/실행 오류 없음 | |
| 핵심 기능 동작 | |
| 에러 처리 기본 구현 | |
| 코드 가독성 (명명, 구조) | |
| SPEC 범위 이탈 없음 | |

### Phase 3: 최종 판정

**PASS**: 모든 AC 통과 + 품질 심사 4/5 이상
→ `개발 완료. 배포 또는 다음 단계 진행 가능.`

**CONDITIONAL PASS**: AC 전부 통과 + 품질 심사 3/5
→ `기능은 완성. 품질 개선 권장 항목: [목록]`

**FAIL**: AC 미통과 항목 존재
→ `/dev-loop를 다시 실행하세요. 미통과 AC: [목록]`

### Phase 4: 결과 보고

1. 터미널에 상세 결과 출력
2. 텔레그램으로 최종 판정 결과 전송:
   ```
   [dev-verify 결과]
   판정: PASS / CONDITIONAL PASS / FAIL
   AC 통과: N/N
   품질: N/5
   ```

## 사용 예시

```
/dev-verify
/dev-verify --strict   (CONDITIONAL PASS를 FAIL로 처리)
```
