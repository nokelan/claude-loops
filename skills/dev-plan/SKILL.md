# /dev-plan — 요구사항 인터뷰 & 명세 확정

개발 착수 전 모호한 요구사항을 소크라테스식 질문으로 명확화하고, 실행 가능한 SPEC.md를 생성한다.

## 핵심 원칙

- 가정을 세우지 않는다. 불명확한 것은 반드시 질문한다.
- 완료 기준(Acceptance Criteria)이 없으면 SPEC.md를 생성하지 않는다.
- SPEC.md가 존재해야만 /dev-loop를 실행할 수 있다.

## 실행 흐름

### Phase 1: 목표 파악

사용자 입력을 받아 다음을 파악한다:
- 무엇을 만들려는가? (기능 목적)
- 누가 사용하는가? (사용자/컨텍스트)
- 어떤 기술 스택을 사용하는가? (또는 제약 없으면 추천)
- 언제까지 완성이 필요한가? (범위 제한)

### Phase 2: 모호성 제거 인터뷰

다음 항목이 불명확하면 한 번에 최대 3개씩 질문한다:

**기능 관련:**
- 핵심 기능은 정확히 무엇인가? (MVP 범위)
- 제외할 기능은 무엇인가?
- 엣지 케이스가 있는가?

**기술 관련:**
- 기존 코드베이스가 있는가? (있으면 경로 확인)
- 외부 API/DB 연동이 필요한가?
- 배포 환경은 어디인가?

**완료 기준:**
- 무엇을 보면 "완성됐다"고 판단하는가?
- 테스트 방법은 무엇인가? (실행해서 확인 / 자동 테스트 / 화면 확인)

모호성이 충분히 해소되면 Phase 3으로 진행. 사용자에게 "이대로 진행할까요?" 확인 후 넘어간다.

### Phase 3: SPEC.md 생성

프로젝트 루트에 `SPEC.md`를 생성한다. 형식:

```markdown
# [프로젝트명] SPEC

## 목표
[한 문장으로 무엇을 만드는지]

## 기능 범위
### 포함
- [기능 1]
- [기능 2]

### 제외
- [명시적으로 제외한 것]

## 기술 스택
- 언어/프레임워크: 
- DB: 
- 배포: 

## 완료 기준 (Acceptance Criteria)
- [ ] AC-1: [구체적 검증 가능한 기준]
- [ ] AC-2: [구체적 검증 가능한 기준]
- [ ] AC-3: [구체적 검증 가능한 기준]

## 작업 목록
1. [구현 단위 태스크 1]
2. [구현 단위 태스크 2]
3. ...

## 메모
[특이사항, 레퍼런스, 주의사항]
```

### Phase 4: 완료 보고

- SPEC.md 생성 완료를 터미널과 텔레그램으로 보고한다.
- 다음 단계 안내: `/dev-loop 로 개발을 시작하세요.`

## 사용 예시

```
/dev-plan 할일 관리 웹앱 만들어줘
/dev-plan
```
