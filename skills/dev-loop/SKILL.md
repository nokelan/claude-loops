# /dev-loop — 구현 + 검증 자동 루프

SPEC.md의 완료 기준(AC)을 통과할 때까지 구현→검증→수정 루프를 자동으로 반복한다.

## 핵심 원칙

- SPEC.md가 없으면 즉시 중단하고 `/dev-plan 을 먼저 실행하세요.` 안내
- 각 루프 사이클마다 텔레그램으로 진행 상황 보고
- 루프는 최대 5회 반복. 5회 초과 시 중단하고 사용자에게 개입 요청
- 수정은 실패한 AC 항목에 한정. 다른 부분 건드리지 않음

## 실행 흐름

### Phase 0: SPEC.md 확인

1. 프로젝트 루트에서 `SPEC.md` 파일을 읽는다.
2. 없으면: `SPEC.md가 없습니다. /dev-plan을 먼저 실행하세요.` 출력 후 종료.
3. 있으면: 완료 기준(AC 목록)과 작업 목록을 추출하여 내부에 저장.
4. 텔레그램으로 루프 시작 알림 전송 (AC 목록 포함)

### Phase 1: 구현

작업 목록의 각 태스크를 순서대로 구현한다.
- 각 태스크 완료 시 간단한 진행 로그 출력
- 파일 변경 내역을 루프 사이클 일지에 기록

### Phase 2: 검증 (3단계)

#### Stage 1 — 기계적 검증 (빌드/실행 확인)
- 빌드가 성공하는가? (`npm run build`, `python -m py_compile`, 등)
- 실행이 되는가? (서버 기동, CLI 실행 등)
- 오류 없이 돌아가는가?

Stage 1 실패 → Phase 3(수정)으로 즉시 이동

#### Stage 2 — 기능 검증 (AC 체크)
SPEC.md의 각 AC 항목을 하나씩 확인한다:
- 코드를 실행/테스트하여 AC가 충족되는지 검사
- 충족: `[PASS] AC-N: ...`
- 미충족: `[FAIL] AC-N: ...` + 실패 원인 기록

Stage 2에서 FAIL 항목 존재 → Phase 3(수정)으로 이동

#### Stage 3 — 의도 검증 (목표 정합성)
- 구현이 SPEC의 "목표" 섹션과 일치하는가?
- 범위 외 기능이 추가되지 않았는가?
- 제외 기능이 실수로 구현되지 않았는가?

Stage 3 실패 → Phase 3(수정)으로 이동

**모든 Stage 통과 → Phase 4(완료)**

### Phase 3: 수정 (루프)

루프 카운터 증가 (최대 5회)

1. 실패한 AC/오류 목록을 분석
2. 원인을 파악하고 수정 계획 수립
3. 수정 범위를 FAIL 항목에만 한정하여 코드 수정
4. 텔레그램으로 수정 내용 보고: `[루프 N/5] 수정 중: ...`
5. Phase 2(검증)로 돌아가기

루프 5회 초과 시:
```
루프 한도 초과 (5회). 자동 해결 불가.
실패 항목: [목록]
사용자 개입이 필요합니다.
```
텔레그램으로 알림 전송 후 종료.

### Phase 4: 완료

모든 AC 통과 시:

1. SPEC.md의 AC 체크박스를 `[x]`로 업데이트
2. 완료 보고서 출력:
   ```
   개발 루프 완료
   총 루프 사이클: N회
   통과 AC: N/N
   소요 시간: 약 N분
   ```
3. 텔레그램으로 완료 알림 전송
4. 다음 단계 안내: `/dev-verify 로 최종 검증을 진행하세요.`

## 사용 예시

```
/dev-loop
/dev-loop --max-loops 3
```
