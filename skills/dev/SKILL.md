# /dev — 개발 루프 오케스트레이터

dev-plan → dev-loop → dev-verify를 순서대로 자동 실행하며 완료 기준 통과까지 전체 파이프라인을 조율한다.

## 오케스트레이터 역할

이 스킬은 직접 코드를 짜지 않는다. 각 단계 스킬을 순서대로 실행하고, 결과를 보고 다음 단계로 넘길지 판단한다.

```
사용자 요청
    ↓
[Phase 1] /dev-plan  — 요구사항 인터뷰 + SPEC.md 생성
    ↓ (SPEC.md 완성 확인)
[Phase 2] /dev-loop  — 구현 + 검증 루프 (최대 5회)
    ↓ (모든 AC 통과 확인)
[Phase 3] /dev-verify — 최종 품질 심사
    ↓
완료 보고 (텔레그램)
```

## 실행 흐름

### 시작

사용자 입력에서 개발 목표를 추출한다.
- `/dev 할일 관리 앱 만들어줘` → 목표: "할일 관리 앱"
- `/dev` (목표 없음) → 무엇을 만들지 먼저 질문

텔레그램으로 파이프라인 시작 알림 전송:
```
[dev 파이프라인 시작]
목표: {목표}
단계: plan → loop → verify
```

### Phase 1: dev-plan 실행

1. `/dev-plan` 스킬을 목표와 함께 실행한다.
2. SPEC.md가 생성될 때까지 대기한다.
3. SPEC.md 존재 확인 후 내용 검토:
   - 완료 기준(AC)이 명확한가?
   - 작업 목록이 구체적인가?
   - 부족하면 dev-plan에 보완 요청
4. SPEC.md 확정되면 Phase 2로 이동.

### Phase 2: dev-loop 실행

1. `/dev-loop` 스킬을 실행한다.
2. 루프 사이클마다 텔레그램 보고를 수신하며 진행 상황 모니터링.
3. dev-loop 종료 후 결과 확인:
   - **모든 AC 통과** → Phase 3으로 이동
   - **루프 한도 초과(5회)** → 사용자에게 개입 요청 후 대기:
     ```
     [파이프라인 중단]
     자동 수정 한도 초과. 수동 개입 필요.
     문제: {실패한 AC 목록}
     수정 후 /dev-loop 를 다시 실행해 주세요.
     ```
   - **사용자가 재개 신호** → Phase 2 재시작

### Phase 3: dev-verify 실행

1. `/dev-verify` 스킬을 실행한다.
2. 판정 결과 확인:
   - **PASS** → 최종 완료 처리
   - **CONDITIONAL PASS** → 권장 개선사항 보고 후 완료 처리
   - **FAIL** → Phase 2(dev-loop)로 되돌아가서 재실행

### 완료

모든 단계 통과 시 최종 보고:

터미널:
```
================================================
개발 파이프라인 완료
목표: {목표}
SPEC.md: {경로}
최종 판정: PASS / CONDITIONAL PASS
총 루프 사이클: N회
================================================
```

텔레그램:
```
[dev 파이프라인 완료]
목표: {목표}
판정: PASS
AC 통과: N/N
총 루프: N회
```

## 중간 개입 포인트

오케스트레이터는 다음 시점에 사용자 확인을 받는다:
- Phase 1 완료 후: "SPEC.md 검토해주세요. 이대로 구현할까요?"
- Phase 2 루프 한도 초과 시: 수동 개입 요청

그 외에는 자율 실행한다.

## 사용 예시

```
/dev 할일 관리 웹앱 만들어줘
/dev 기존 프로젝트에 로그인 기능 추가
/dev
```
